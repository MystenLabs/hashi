OperatorInitRequest
Operator-supplied bootstrap. A ceremony-mode enclave (setup/rotate) needs only s3_config; a withdraw-mode enclave additionally carries the WithdrawModeConfig (committee, limiter, BTC master pubkey, secret-sharing instance, network) whose digest is the share-decryption AAD.
ProvisionerInitRequest
The current KPs’ encrypted key shares, assembled into one submission (in the relay model, by the relay once it has collected enough). Each share’s HPKE AAD binds the enclave’s state_hash (the WithdrawModeState digest), so a share only decrypts if the KP agreed on the operator-supplied state.
RotateKpsRequest
Ceremony-mode rotation request, assembled by the operator from the current KPs’ encrypted old shares (each bound to state.digest() as AAD) and the shared rotation target state.
RotateKpsResponse

RotateKpsState
The shared rotation target all current KPs authorize. Each binds state.digest() as HPKE AAD on its submission, so the enclave only decrypts ones that agree on it. Old/new (n, t) may differ.

WithdrawModeConfig
Full operator-supplied withdraw-mode config: the attested WithdrawModeState plus delivery-only fields that are enforced elsewhere and so are excluded from the digest (the instance via direct share verification; network is share-irrelevant). Supplied to operator_init.
WithdrawModeState
The withdraw-mode state KPs attest to. Its digest() is the state_hash: bound as HPKE AAD on each KP’s share and exposed (as a hash) via GuardianInfo. These are exactly the fields whose agreement is enforced only via the digest.

Enums§

EnclaveMode
Which flows an enclave serves, fixed at boot. A Ceremony enclave runs setup_new_key/rotate_kps; a Withdraw enclave runs provisioner_init + standard_withdrawal. operator_init, get_guardian_info are enabled in both modes.

SESSION_ID_HEX_LEN
