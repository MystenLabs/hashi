tr(NUMS, {multi_a(2, guardian_btc_pubkey, h), and_v(v:older(delay), pk(h))}) where h = derive(mpc_pubkey, path): an immediate guardian+MPC 2-of-2 leaf, plus an MPC-only recovery leaf spendable after HASHI_MPC_RECOVERY_DELAY_SECONDS. path = None (change address) maps to a zero-byte path so deposit and withdrawal sides agree on the leaf key without a special case.

mpc_key is the raw MPC verifying key (G). The derivation is taken directly against this point — using only the x-only projection would silently force the parent to even-y and produce a different child key for ~half of all MPC vks, breaking the 2-of-2 leaf script.
